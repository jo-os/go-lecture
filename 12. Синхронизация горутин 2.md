## 1. Инструменты синхронизации в Go
В предыдущих разделах мы рассмотрели ошибки, которые могут возникнуть в многопоточном (точнее — многогорутичном) коде и как они влияют на стабильность, производительность и предсказуемость работы программы. Теперь пришло время разобраться, какие средства предоставляет Go для синхронизации доступа к разделяемым данным и управления конкуренцией.

Go предлагает несколько ключевых инструментов, каждый из которых предназначен для определённых задач:
```
sync.WaitGroup
sync.Mutex
sync.RWMutex 
sync/atomic
```
Эти средства покрывают большую часть практических сценариев синхронизации и позволяют писать безопасный и предсказуемый конкурентный код.

### WaitGroup — координация горутин
**Зачем нужен WaitGroup?**
Когда вы запускаете несколько горутин, может понадобиться дождаться их завершения. Это особенно важно, если горутины выполняют обработку данных перед переходом к следующему этапу.

**Как работает WaitGroup?**
Он отслеживает количество активных горутин и ожидает их завершения через метод Wait().

**Основные методы:**
- Add(n int) — увеличивает счётчик на n.
- Done() — уменьшает счётчик на 1 (обычно вызывается через defer).
- Wait() — блокирует выполнение до тех пор, пока счётчик не станет равным нулю.

**Пример:**
```go
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
wg.Add(1)
go func(id int) {
defer wg.Done()
fmt.Printf("Горутина %d завершила работу\n", id)
}(i)
}
wg.Wait()
fmt.Println("Все горутины завершены")
```
**На что обратить внимание:**
- Не вызывайте Add() после Wait() — это может привести к гонке данных.
- Если Done() не будет вызван, Wait() зависнет навсегда.

### Mutex — защита разделяемых данных
Зачем нужен Mutex?
Когда несколько горутин обращаются к одним и тем же переменным, необходимо синхронизировать доступ, чтобы избежать гонки данных.

Тип: sync.Mutex

**Методы:**
- Lock() — захватывает блокировку.
- Unlock() — освобождает блокировку.

**Пример:**
```go
var (
counter int
mu sync.Mutex
)
for i := 0; i < 100; i++ {
go func() {
mu.Lock()
counter++
mu.Unlock()
}()
}
```
**Рекомендации:**
- Всегда вызывайте Unlock() после Lock(). Используйте defer mu.Unlock().
- Не вызывайте Unlock() без Lock() — будет panic.
- Не захватывайте Lock() в одной горутине и не отпускайте его в другой.

**Типичная ошибка:**
```go
mu.Lock()
// ... потенциально длительная операция
mu.Unlock()
```
Критическая секция должна быть как можно короче, чтобы не блокировать другие горутины.

### RWMutex — для частого чтения и редкой записи
sync.RWMutex позволяет:
- нескольким горутинам одновременно читать (RLock),
- только одной — записывать (Lock).

**Методы:**
- RLock() / RUnlock() — блокировка для чтения;
- Lock() / Unlock() — блокировка для записи.

**Пример:**
```go
var (
data = make(map[string]string)
mu sync.RWMutex
)
func readValue(key string) string {
mu.RLock()
defer mu.RUnlock()
return data[key]
}
func writeValue(key, value string) {
mu.Lock()
defer mu.Unlock()
data[key] = value
}
```
RWMutex особенно полезен, если чтение значительно чаще записи.

### atomic — атомарные операции
**Зачем нужен пакет sync/atomic?**
Если нужно изменять значения примитивных типов (например, int64) из нескольких горутин без использования Mutex — для повышения производительности.

**Пример:**
```go
package main
import (
"fmt"
"sync/atomic"
)
func main() {
var counter int64
for i := 0; i < 1000; i++ {
go func() {
atomic.AddInt64(&counter, 1)
}()
}
}
```
**Популярные функции:**
- AddInt64, AddUint32 — атомарное сложение;
- LoadInt64, StoreInt64 — чтение и запись;
- CompareAndSwapInt64 — атомарное сравнение и замена (CAS).

**Пример CAS:**
```
var value int32 = 0
if atomic.CompareAndSwapInt32(&value, 0, 1) {
fmt.Println("Значение обновлено с 0 на 1")
}
```
**Важно:**
- atomic работает только с указателями.
- Операции вроде x++ — неатомарны.
- atomic эффективен, но не подходит для сложных структур.

**Таблица сравнения инструментов**
|Инструмент|Назначение|Пример|Преимущества|Ограничения|
|----------|----------|------|------------|-----------|
|WaitGroup|Ожидание завершения горутин|Параллельная обработка|Простой, понятный интерфейс|Не синхронизирует данные|
|Mutex|Взаимное исключение|Общие переменные|Универсален|Возможен deadlock|
|RWMutex|Оптимизация чтения|Кеш, конфигурации|Высокая производительность при чтении|Не защищает от логических ошибок|
|atomic|Атомарные операции с примитивами|Счётчики, флаги|Максимальная производительность|Подходит только для простых типов|

**Какой инструмент выбрать?**

- Нужно дождаться завершения всех горутин? → WaitGroup
- Нужно синхронизировать доступ к переменной? → Mutex или atomic
- Чтений больше, чем записей? → RWMutex
- Нужна производительность и данные — простые? → atomic
- Требуется управлять жизненным циклом горутин? → Используйте Context

## 2. Практический кейс: устранение гонки данных
В этом кейсе показан пример некорректного конкурентного кода с гонкой данных. Затем мы пошагово разберём, как устранить проблему с помощью инструментов синхронизации, доступных в Go.

Проблема: гонка данных при подсчёте

Предположим, у вас есть задача — запустить 1000 горутин, каждая из которых увеличивает общий счётчик. Наивная реализация может выглядеть так:
```go
package main
import (
"fmt"
)
func main() {
counter := 0
for i := 0; i < 1000; i++ {
go func() {
counter++
}()
}
fmt.Println("Результат:", counter)
}
```
Что здесь не так:
- counter++ — неатомарная операция. Она состоит из чтения, увеличения и записи, которые могут прерываться другими горутинами.
- Программа не ожидает завершения горутин — скорее всего, fmt.Println выполнится раньше, чем счётчик изменится.

Возможный результат:
0, 1, 27, 153... но почти никогда 1000. Это типичное проявление гонки данных.

### Решение 1: WaitGroup + Mutex
Используем:
- sync.WaitGroup — чтобы дождаться завершения всех горутин;
- sync.Mutex — для безопасного изменения переменной counter.
```go
package main
import (
"fmt"
"sync"
)
func main() {
var counter int
var mu sync.Mutex
var wg sync.WaitGroup
for i := 0; i < 1000; i++ {
wg.Add(1)
go func() {
defer wg.Done()
mu.Lock()
counter++
mu.Unlock()
}()
}
wg.Wait()
fmt.Println("Результат:", counter)
}
```
Как работает:
- wg.Add(1) добавляет горутину в счётчик ожидания;
внутри горутины:
- defer wg.Done() — сообщает о завершении;
- mu.Lock() и mu.Unlock() — защищают counter;
- wg.Wait() — блокирует основную горутину до завершения всех остальных.

Результат: counter == 1000, предсказуемо и корректно.

### Решение 2: атомарная операция
Если нужно просто увеличивать числовое значение — удобнее использовать sync/atomic.
```go
package main
import (
"fmt"
"sync"
"sync/atomic"
)
func main() {
var counter int64
var wg sync.WaitGroup
for i := 0; i < 1000; i++ {
wg.Add(1)
go func() {
defer wg.Done()
atomic.AddInt64(&counter, 1)
}()
}
wg.Wait()
fmt.Println("Результат:", counter)
}
```
Пояснение:
- atomic.AddInt64 — атомарно увеличивает значение без Mutex;
- переменная counter должна быть строго int64 или совместимым типом;
- atomic эффективнее при большом числе операций.

### Проверка гонки: флаг -race
Запускайте программу с флагом -race:
```
go run -race main.go
```
В исходной (неправильной) версии будет предупреждение о гонке. 

В исправленных примерах — ошибок не будет.

### Вывод
Параллельный доступ к данным требует явной синхронизации. Без неё возникают гонки данных и непредсказуемое поведение.

Инструменты Go для безопасной работы с конкурентностью:
|Инструмент|Назначение|
|----------|----------|
|WaitGroup|Ожидание завершения горутин|
|Mutex|Защита разделяемых данных|
|atomic|Атомарные операции с простыми типами|

Подведём итоги.

Что разобрали:
- Синхронизация — необходимость. Без неё горутины могут одновременно менять одни и те же данные. Это приводит к непредсказуемому поведению, потерям данных и сбоям. Горутины не гарантируют порядок выполнения — его нужно контролировать вручную.

- Ошибки конкурентного кода бывают разными:
    - Гонки данных (data races) — доступ к переменной без синхронизации
    - Взаимные блокировки (deadlocks) — горутины навсегда ждут друг друга.
    - Утечки горутин (goroutine leaks) — незавершённые и неиспользуемые горутины продолжают потреблять ресурсы.
    - Голодание (starvation) — задачи не получают доступ к ресурсам из-за активной загрузки другими.
- Последствия ошибок конкурентности — коварны. Они часто не проявляются в тестах и возникают только в условиях высокой нагрузки или в продакшене. Программа может «работать» — до тех пор, пока не сломается в самый неподходящий момент.
- Go предоставляет эффективные инструменты для синхронизации:
    - sync.WaitGroup — для ожидания завершения группы горутин;
    - sync.Mutex и sync.RWMutex — для безопасного доступа к разделяемым данным;
    - sync/atomic — для атомарных операций с простыми типами данных.
