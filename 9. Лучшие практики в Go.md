# Зачем придерживаться лучших практик в Go
Когда вы только начинаете писать код, главное — чтобы программа работала. Это естественно. Однако со временем задачи усложняются, а над проектами начинают работать команды. В таких условиях становится критически важно писать код так, чтобы его было удобно читать, поддерживать и развивать.
 
Представьте ситуацию:
Вы создаёте небольшой проект — 10 файлов, 2 000 строк. Всё ясно — вы только что это написали. Через месяц к проекту подключается другой разработчик. А ещё через полгода вы сами возвращаетесь к этому коду. И тут становится очевидно: даже собственный код может оказаться нечитаемым, непредсказуемым и неудобным, если писать его без системы.
 
Хорошие практики — это не про скучные правила ради галочки. Это про то, чтобы:
- вам было проще ориентироваться в собственном коде;
- ваши коллеги не тратили время на «разгадывание» логики;
- программа реже «падала» и проще обновлялась;
- команда могла расти без потери качества — когда все пишут в одном стиле, проект развивается быстрее.
 
Писать понятно — значит уважать своё время и время команды.
 
**Чем особенно важны практики в Go**
 
Go создавался с акцентом на простоту, предсказуемость и поддержку кода. Это язык, в котором уже заложены чёткие рекомендации по стилю и архитектуре. Их не нужно придумывать — они есть и активно применяются сообществом.
 
Следуя этим рекомендациям, вы быстрее становитесь эффективным разработчиком. Ваш код легче интегрируется в командную работу и вызывает доверие у коллег.
 
## Именование переменных, функций и пакетов
Имена — первое, что видит другой разработчик, читая ваш код. Даже если логика работает, плохо выбранные имена могут сбить с толку. Хорошее имя — это уже половина понимания кода.
 
**Основной принцип: названия должны «говорить»**
 
Если по имени непонятно, что делает переменная, функция или пакет — это плохое имя. Хорошее имя объясняет себя само, без дополнительных комментариев.
 
**Как давать хорошие имена:**
 
- Используйте понятные и конкретные слова
    - Примеры: userName, productList, isActive

- Избегайте аббревиатур, если они не общеизвестны
    - Пример: cfg (для конфигурации) — допустимо, pnl (для "panel" или "profit and loss") — уже спорно.

- Не сокращайте слова до одной-двух букв, если только это не временные переменные в коротком контексте
    - Пример: a, x1, tmp — допустимо в пределах цикла, но не во всей функции.

- Функции называются глаголами, если они выполняют действия
    - Примеры: sendEmail, loadUser.

- Пакеты называются короткими существительными в единственном числе
    - Примеры: math, user, storage.

**Плохой пример:**
```go
func d(x int) int {
    return x * x
}
```
Неясно, что делает функция, что такое x, зачем она нужна.
 
**Хороший пример:**
```go
func square(number int) int {
    return number * number
}
```
Понятно, что функция возвращает квадрат числа. Переменная имеет описательное имя.
 
**Плохой пример (функция с побочным действием):**
```go
func data(user string) {
// ...
}
```
Имя функции не содержит действия. Что она делает? Получает данные? Удаляет?
 
**Хороший пример:**
```go
func saveUserData(username string) {
// ...
}
```
Сразу ясно — функция сохраняет данные пользователя. Имя переменной также понятно.
 
**Плохой пример (пакет):**
```
import "myapp/processingdata"
```
Длинное и не по-Go. Неясно, что внутри — и сложно писать processingdata.DoSomething().
 
**Хороший пример:**
```go
import "myapp/process"
```
Коротко и ясно. Имя пакета видно в коде, его удобно использовать. 

### Советы из практики:
- Представляйте, что ваш код будет читать другой человек. В идеале — не знакомый с контекстом.
- Не экономьте символы. totalPrice лучше, чем tp.
- Имена — это инвестиция в будущее. Плохие названия сложнее объяснять, чем поменять сразу.
- Если не удаётся придумать имя сразу, попробуйте вслух описать, что делает объект. Имя придёт в процессе.

## Форматирование кода с помощью gofmt
Когда над проектом работают несколько человек, у каждого может быть свой стиль оформления кода: кто-то ставит пробел перед скобкой, кто-то делает отступы в 3 пробела, кто-то выравнивает переменные по колонкам. На работу программы это не влияет, но мешает чтению и поддержке кода.
 
В Go форматирование — автоматическое, то есть инструмент gofmt входит в стандартную поставку Go. Он автоматически приводит код к единому стилю, принятому во всём сообществе. Это избавляет от споров и субъективных предпочтений.
 
Вы просто запускаете команду — и получаете аккуратно оформленный код.
```
Как пользоваться gofmt
Из терминала вручную: gofmt -w main.go

Флаг -w означает, что отформатированный код будет записан прямо в файл.

Автоматически при сохранении файла:

Во всех Go-проектах принято настраивать редактор (например, VS Code, GoLand) так, чтобы gofmt применялся при каждом сохранении. Это делается один раз и позволяет не думать о форматировании вообще.
```
До форматирования:
```go
func add (a int,b int)int{
return a+b}
```
Код сложно читать: отступы нарушены, пробелы расставлены хаотично.
 
После gofmt:
```go
func add(a int, b int) int {
    return a + b
}
```
Код стал понятным, структурированным и единообразным.
 
**Почему важно использовать gofmt всегда:**
 
Экономит время: не нужно вручную наводить порядок в коде.
Убирает споры. Все пишут в одном стиле — потому что так настроено.
Повышает читаемость. Аккуратный код проще понять и сопровождать.
Проходит автоматические проверки. Многие CI-инструменты (например, GitHub Actions, GitLab CI) не примут плохо отформатированный код.

## Документирование кода: комментарии и godoc
Даже хорошо подобранные имена переменных и функций не всегда объясняют, почему в коде принято то или иное решение. Особенно это важно в командной работе, когда один человек пишет код, а другой его читает, редактирует или дополняет.
 
Go предлагает простой и удобный способ документирования кода, который работает не только для чтения человеком, но и для автоматической генерации документации через инструмент godoc .
 
Главное правило Go: комментарий начинается с имени объекта
 
Если вы комментируете функцию, тип или метод, начинайте комментарий с его имени. Такой формат распознаётся godoc и включается в автодокументацию.
 
Для экспортируемых элементов (то есть тех, чьи имена начинаются с заглавной буквы) комментарии обязательны. Это позволяет другим разработчикам понять, что делает функция, не заглядывая в реализацию.
 
**Пример хорошего комментария:**
```go
// Add возвращает сумму двух целых чисел.
func Add(a, b int) int {
return a + b
}
```
Комментарий начинается с имени функции, кратко и точно описывает её назначение.
 
**Плохой пример:**
```go
// Эта функция делает сложение
func Add(a, b int) int {
return a + b
}
```
Комментарий не начинается с имени функции и даёт слишком общее описание. Непонятно, какие данные складываются и что именно возвращается.
 
**Комментарии внутри кода**
 
Внутри функций комментарии используются для пояснения неочевидной логики или важных особенностей.
 
**Пример уместного комментария:**
```go
func calculateDiscount(price float64) float64 {
if price > 100 {
// Скидка 10% применяется только к товарам дороже 100
    return price * 0.9
    }
return price
}
```
Комментарий объясняет деловую логику: почему скидка применяется только при определённой цене.
 
**Пример лишнего комментария:**
```go
func double(x int) int {
// Умножаем x на 2
return x * 2
}
```
Комментарий повторяет очевидное. Такие пояснения не несут ценности и только засоряют код.
 
**Практические советы:**
- Комментарий не должен дублировать код. Он должен дополнять его — объяснять почему, а не как.
- Объясните неочевидную логику. Особенно если она зависит от бизнес-правил или нестандартных решений.
- Следит за актуальностью. Устаревший комментарий может ввести в заблуждение — это хуже, чем его отсутствие.
- Следуйте стилю Go:
```
// ИмяФункции делает то-то… — такой комментарий попадёт в godoc без лишних усилий.
```
## Тестирование: как проверять, что код работает правильно
Тесты нужны не только для поиска ошибок. Они помогают:
- точно определить, что должен делать код;
- убедиться, что после изменений ничего не сломалось;
- находить баги без запуска всей программы вручную.
 
**В Go всё просто: тесты — это обычные функции**
 
Go встроенно поддерживает юнит-тестирование. Вам не нужно подключать отдельные библиотеки — всё уже есть в стандартной библиотеке.
 
Чтобы написать тест, нужно:
```
Создать файл с именем, оканчивающимся на _test.go.
Написать функцию, имя которой начинается с Test и аргументом принимает *testing.T.
```
**Пример базового теста**
```go
// Функция, которую мы хотим протестировать
func Add(a, b int) int {
    return a + b
}
// Файл: add_test.go
import "testing"

func TestAdd(t *testing.T) {
    result := Add(2, 3)
    expected := 5
    if result != expected {
        t.Errorf("Ожидали %d, но получили %d", expected, result)
    }
}
```
Если функция Add(2, 3) возвращает не 5 — тест завершится с ошибкой и покажет сообщение.
 
**Плохой пример**
```go
func TestAdd(t *testing.T) {
    Add(2, 3)
}
```
Тест не проверяет результат. Он ничего не сообщает при ошибке и всегда "проходит". Такой тест — бесполезен.
```
Как запускать тесты
В терминале, из каталога проекта: go test
```
Команда автоматически найдёт все файлы с именем *_test.go, выполнит тесты и выведет результат:
- Если тесты прошли — вы увидите ok.
- Если есть ошибки — будет показано, что именно не совпало.
 
**Советы из практики:**
- Пишите тесты сразу, вместе с функцией — это помогает лучше формулировать поведение.
- Один тест — один случай. Лучше несколько простых тестов, чем один громоздкий.
- Используйте t.Errorf, чтобы объяснять, что пошло не так.
- Помните: тесты — это тоже код. Пишите их понятно и лаконично.

## Обработка ошибок: как писать надёжный код
Go — это язык про простоту, предсказуемость и надёжность. Одна из его ключевых особенностей — явная обработка ошибок. В отличие от языков с исключениями, в Go ошибки возвращаются как значения, и разработчик обязан сам их обработать.
 
Это может показаться неудобным, но такой подход делает ошибки предсказуемыми и контролируемыми.
 
**Как устроена ошибка в Go**
 
Функции, которые могут завершиться с ошибкой, обычно возвращают два значения: результат и ошибку. Ошибку нужно проверять сразу после вызова.

**Пример корректной обработки ошибки:**
```go
data, err := os.ReadFile("config.txt")
if err != nil {
    log.Fatalf("не удалось прочитать файл: %v", err)
}
fmt.Println(string(data))
```
Код не продолжает выполнение, если произошла ошибка — это безопасно и прозрачно.
 
**Плохой пример: игнорирование ошибки**
```go
data, _ := os.ReadFile("config.txt")
fmt.Println(string(data))
```
Использование _ вместо проверки err означает: «мне не важна ошибка». Это опасно — программа продолжит работу даже если файл не существует, и может завершиться с ошибкой позже, в менее очевидном месте.
 
**Что делать, если ошибка есть?**
 
Go не выбрасывает исключения. Вместо этого вы сами решаете, как обрабатывать ошибку:
- вывести сообщение пользователю;
- вернуть ошибку дальше;
- остановить выполнение программы (в особых случаях — с помощью panic);
- попытаться восстановиться.
 
**Пример: возврат ошибки вызывающему коду**
```go
func loadConfig(path string) ([]byte, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("ошибка при загрузке %s: %w", path, err)
    }
    return data, nil
}
```
Функция не завершает программу, а добавляет контекст к ошибке и возвращает её выше — вызывающий код сможет решить, как с ней поступить.
 
**Про panic (panic используется только в крайних случаях):**
- нарушение инвариантов (например, деление на ноль, обращение к nil);
- баги в библиотеке;
- состояние, из которого невозможно продолжать выполнение.
 
В обычной бизнес-логике вручную вызывать panic не следует.
 
**Советы по работе с ошибками:**
- Всегда проверяйте err после вызова функции, которая может вернуть ошибку.
- Не бойтесь возвращать ошибку — это нормальная практика в Go.
- Добавляйте контекст с помощью fmt.Errorf("что-то пошло не так: %w", err).
- Используйте log.Fatalf() или log.Println(), если ошибка критична.
- Никогда не игнорируйте ошибку, даже временно. Не используйте _ вместо err, если только вы не уверены в безошибочности функции (и то — с осторожностью).

## Организация кода по пакетам и файлам
Когда программа состоит из одного файла — всё просто. Но по мере роста проекта код становится объёмнее и сложнее. Если всё держать в одном месте, будет трудно:
- понять, где находится нужная часть кода;
- повторно использовать уже написанное;
- вносить изменения без риска повредить что-то другое.
 
Решение — грамотное разбиение кода на пакеты и логические файлы.

**Что такое пакет?**
 
Пакет (package) в Go — это группа связанных файлов, собранных в одну папку. Каждый файл в пакете должен начинаться с одной и той же строки:
```
package имя
```
Имя пакета, как правило, совпадает с названием папки. Из других частей программы можно импортировать пакет и использовать его экспортируемые функции.
 
**Пример структуры проекта**
```
myapp/
├── main.go // вход в программу
├── user/
│ └── user.go// логика, связанная с пользователями
├── math/
│ └── sum.go // арифметические функции
```
```go
main.go:

package main
import (
    "fmt"
    "myapp/math"
)

func main() {
    fmt.Println(math.Sum(2, 3))
}
```
```go
math/sum.go:
package math
func Sum(a, b int) int {
    return a + b
}
```
Пакет math содержит функцию Sum, которую можно вызывать из других пакетов, например, из main.
 
**Плохой пример (всё в одном файле):**
```go
package main
import "fmt"
func Sum(a, b int) int {
    return a + b
}

func main() {
    fmt.Println(Sum(2, 3))
}
```
Такой код работает, но только пока проект маленький. По мере роста:
- логика перемешивается;
- файл становится громоздким;
- сопровождение усложняется.

Принципы организации кода:
- Один пакет — одна ответственность.
    - Например: user, auth, storage.
- Использование пакетов — короткие и конкретные.
    - Используются существительные, без глаголов: config, db, email.
- В пакете может быть несколько файлов.
    - Не нужно всё складывать в один .go файл.
- main.go — точка входа.
    - Только запуск программы. Логику стоит выносить в отдельные пакеты.
- Экспортируемые функции и переменные начинаются с заглавной буквы.
    - Всё, что начинается с маленькой буквы — доступно только внутри пакета.
