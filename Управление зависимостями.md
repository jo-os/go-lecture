# Управление зависимостями
### Почему иногда стандартной библиотеки не хватает
Когда вы только начинаете писать на Go, всё кажется простым. Один файл, main(), пара строк — и программа уже работает. Go действительно славится лаконичностью, а его стандартная библиотека покрывает многие базовые задачи: чтение и запись файлов, работа с сетью, форматирование текста.
 
Но со временем задачи усложняются. Например, нужно:
- сгенерировать уникальный ID для пользователя;
- отправить HTTP-запрос с тайм-аутом и распарсить JSON-ответ;
- проверить корректность email-адреса;
- сохранить данные в YAML, а не в JSON.
С такими задачами стандартной библиотеки уже не хватает. Приходится искать сторонние решения — и тут на помощь приходят внешние библиотеки: готовые пакеты, которые пишут и поддерживают другие разработчики.
 
**Пример**
 
Допустим, вы пишете небольшой CLI-инструмент, который записывает события. Каждое событие должно иметь уникальный идентификатор. Как его сгенерировать? Вручную — ненадёжно. Лучше использовать UUID.

В Go нет встроенной функции для генерации UUID, но есть популярные сторонние библиотеки. Вот как может выглядеть ваш код:
```go
package main
import (
    "fmt"
    "uuidlib" // воображаемая внешняя библиотека
)
func main() {
    id := uuidlib.Generate()
    fmt.Println("Сгенерированный UUID:", id)
}
```
Разумеется, uuidlib — это условный пример. Но он показывает главное: иногда вам нужен внешний пакет, который уже решает вашу задачу.

### Что такое зависимости и зачем ими управлять
Когда вы пишете программу на Go, вы почти сразу начинаете использовать чужой код. Даже если программа выглядит очень просто:
```go
package main
import "fmt"

func main() {
    fmt.Println("Привет, Go!")
}
```
Здесь fmt — это не ваш код. Это встроенный пакет из стандартной библиотеки Go. Он предоставляет функции для вывода текста: Println, Printf, Sprintf и другие. То есть вы уже используете зависимость.
### Стандартные зависимости
Go поставляется с мощной стандартной библиотекой. В ней — десятки готовых пакетов для работы со временем, HTTP, JSON, шифрованием и не только:
- fmt — форматированный вывод
- time — работа с датами и временем
- os — доступ к файловой системе и переменным окружения
- strings — работа со строками
- math/rand — генерация случайных чисел
- net/http — клиент и сервер HTTP
 
Эти зависимости гарантированно стабильны — они встроены в Go и обновляются вместе с ним. Вы можете на них рассчитывать.
 
**А если библиотека не встроена?**
 
Как только вы решаете использовать сторонний пакет (например, для работы с YAML или генерации UUID), Go должен:
- понять, где взять исходный код;
- скачать и сохранить его;
- учесть нужную версию;
- обеспечить совместимость с вашей программой.
 
Именно тут появляется задача управления зависимостями. Go должен точно знать:
- какую библиотеку вы подключили;
- какую её версию использовать;
- как другим разработчикам собрать тот же проект без ошибок.
 
**Почему это важно?**
 
Если вы не управляете зависимостями, проект может перестать собираться в любой момент. Например, библиотека обновилась, изменила интерфейс — и ваш код больше не компилируется.
 
Контроль версий нужен, чтобы:
- Сборка была стабильной — независимо от изменений в интернете.
- Версии были зафиксированы — чтобы избежать неожиданностей.
- Команда работала синхронно — у всех одни и те же зависимости.
- Сборка была безопасной — Go не подтянет уязвимую или сломанную версию.
 
**Вывод:** зависимость — это любой внешний по отношению к вашему коду пакет. Даже стандартные библиотеки — это зависимости. Управление ими — это способ сделать ваш проект надёжным, предсказуемым и удобным для команды.
 
### Что такое go mod и как он устроен
По мере роста проектов на Go стало понятно: нужна централизованная система для управления зависимостями. Так появилась система модулей go mod.
 
Она встроена в Go с версии 1.11, а с 1.13 — используется по умолчанию.

### Что такое модуль в Go
Модуль — это просто ваш проект. Логически связанный набор пакетов в одной директории, который можно использовать сам по себе или как зависимость в другом проекте.
 
Модули могут зависеть от других модулей — так строится граф зависимостей. Чтобы управлять этими связями, Go использует два файла:
- go.mod — основной файл модуля;
- go.sum — вспомогательный файл для проверки подлинности зависимостей.
```go
module example.com/myapp
go 1.20
require (
    example.org/lib1 v1.2.3
    example.org/lib2 v0.9.1
)
```
Что значат эти строки:
- module — имя модуля (как его можно будет импортировать).
- go — версия Go, с которой совместим проект.
- require — список внешних библиотек с указанными версиями.
 
go.mod создаётся один раз, но автоматически обновляется при добавлении или удалении зависимостей.
 
**Файл go.sum**
 
Go любит порядок. Поэтому когда вы добавляете внешние библиотеки, он сохраняет контрольные суммы этих библиотек в файл go.sum.

Это нужно для двух вещей:
- Проверка целостности: Go сравнивает хэш и убеждается, что вы используете именно ту версию зависимости, которую указали.
- Повторяемая сборка: если библиотека изменится на сервере, Go это заметит и не даст собрать проект.
 
Совет: всегда добавляйте go.sum в Git, чтобы коллеги могли собрать проект без неожиданностей.
 
### Как Go работает с модулями
Каждый раз, когда вы запускаете go run, go build или go test, Go делает следующее:
- Читает go.mod, чтобы понять, какие библиотеки нужны.
- Проверяет go.sum — всё ли на месте, всё ли по-честному.
- При необходимости скачивает зависимости в кеш ($GOPATH/pkg/mod) — не в ваш проект.
- Собирает проект с нужными версиями библиотек.
 
Go может даже сам добавить нужную зависимость в go.mod, если вы просто написали import и собрали проект. Удобно — но следите за тем, что меняется в go.mod и go.sum.
 
**Резюме**
- go.mod — главный файл, где указаны имя модуля, версия Go и список зависимостей
- go.sum — список контрольных сумм зависимостей для проверки подлинности.
- Go сам управляет версиями, но полагается на точные и актуальные данные в этих файлах.
 
Это и есть основа системы зависимостей Go — прозрачной, воспроизводимой и встроенной в язык.

## Практика: управление зависимостями в Go
Теперь, когда мы понимаем, что такое зависимости и как устроена система go mod, перейдём к практическим действиям. В этой части мы научимся:
- инициализировать модуль;
- добавлять и обновлять зависимости;
- удалять неиспользуемые зависимости.
Всё это — на конкретных примерах с командами и фрагментами кода.

**Инициализация проекта: go mod init**
Чтобы начать работу с зависимостями, проект должен быть модулем. Для этого нужно создать файл go.mod с помощью команды:
```
go mod init myproject
```
Здесь myproject — имя модуля. Оно используется как часть импорта, если ваш код будет использоваться другими.
 
После выполнения этой команды в текущей папке появится файл go.mod. Он пока пустой, кроме строки с именем модуля и версией Go:
```
module myproject
go 1.21
```
Теперь ваш проект стал модулем и готов к подключению зависимостей.
 
**Добавление зависимости: go get или импорт в коде**
 
Есть два способа добавить зависимость.
 
**Способ 1: явно с помощью команды go get**
```
go get example.org/somepkg@v1.2.3
```
После этого Go:
- добавит строку require в go.mod;
- скачает пакет;
- добавит проверку в go.sum.
 
**Способ 2: Импортировать в коде**
 
Если вы просто напишете в коде:
```
import "example.org/somepkg"
```
и затем выполните:
```
go build
```
**Go сам найдёт последнюю подходящую версию этого пакета и обновит go.mod и go.sum.**
 
Пример использования:
```go
package main

import (
    "fmt"
    "example.org/somepkg"
)

func main() {

result := somepkg.DoSomething()
fmt.Println(result)
}
```
После сборки вы увидите, что go.mod пополнился:
```
require example.org/somepkg v1.2.3
```
Обновление зависимости: go get -u
 
Допустим, вы хотите обновить одну конкретную зависимость до более новой версии. Используйте:
```
go get -u example.org/somepkg
```
Если вы хотите обновить все зависимости проекта:
```
go get -u ./...
```
Эта команда просканирует проект и обновит всё, что возможно, до последних версий с сохранением совместимости.

### Удаление зависимости: удаление import + go mod tidy
В Go зависимости удаляются не вручную, а автоматически, если они больше не используются в коде.
 
Удалите все import пакета, который вы хотите убрать.
 
Выполните команду:
- Удалите все import пакета, который вы хотите убрать.
- Выполните команду: go mod tidy
 
Эта команда:
- удалит неиспользуемые зависимости из go.mod;
- очистит go.sum;
- добавит отсутствующие зависимости, если вы забыли использовать go get.
 
Подведём итоги
|Действие|Команда|Комментарий|
|--------|-------|-----------|
|Инициализация модуля|go mod init myproject|Создаёт файл go.mod|
|Добавление пакета|go get example.org/pkg@vX.Y.Z|Добавляет зависимость|
|Импорт в коде|import "example.org/pkg"|Также добавит зависимость после сборки|
|Обновление пакета|go get -u example.org/pkg|Обновляет до новой версии|
|Обновление всего|go get -u ./...|Обновляет все зависимости|
|Удаление|Удалить import, затем go mod tidy|Убирает неиспользуемые зависимости|
 
Таким образом, вы можете полностью контролировать зависимости вашего проекта: добавлять, обновлять, избавляться от лишнего. Всё это — встроено в инструменты Go и работает без дополнительных менеджеров.
 
**Типичные ошибки и полезные команды при работе с зависимостями в Go**
 
Работа с зависимостями — важная, но иногда непростая часть разработки. Ниже разберём частые ошибки и команды, которые помогут держать проект в порядке.
 
Одна из ключевых команд — go mod tidy. Она:
- убирает из go.mod и go.sum пакеты, которые больше не используются;
- добавляет те зависимости, которые используются в коде, но не указаны явно.
 
Запуск команды: go mod tidy
 
Полезна для регулярной «уборки» проекта и предотвращения конфликтов.
 
Чтобы посмотреть, от каких модулей зависит проект, используется: go list -m all
 
Команда выведет список всех зависимостей с указанием версий. Если вы не понимаете, зачем в проекте появилась конкретная библиотека, поможет: go mod why example.org/somepkg Эта команда покажет цепочку импортов, которая приводит к использованию указанного пакета.
 
Теперь о типичных ошибках. Первая — дублирование зависимостей. Это может произойти, если разные библиотеки требуют несовместимые версии одной и той же зависимости. В результате в go.mod появляются конфликты. Чтобы избежать проблем, регулярно запускайте go mod tidy и проверяйте список require.
 
Вторая ошибка — сломанные зависимости. Такое случается, если библиотека была удалена, перемещена или обновилась без обратной совместимости. В этом случае при go build или go get вы получите ошибку. Решение: попробовать обновить пакет через go get -u, либо проверить и настроить replace, если используете локальную копию зависимости.
 
Третья ошибка — отсутствие go.mod. Если проект или его часть не инициализирована как модуль, Go не сможет корректно управлять зависимостями. Каждому проекту (или подпроекту) нужен свой go.mod.
 
Итог: используйте go mod tidy, чтобы наводить порядок, go list — чтобы понимать структуру зависимостей, и go mod why — чтобы быстро находить лишние или неожиданные библиотеки. Зная частые ошибки, вы сможете их избегать и поддерживать проект в стабильном состоянии.

## Выводы
Управление зависимостями — основа стабильной работы над проектами на Go. Вот что важно помнить:
- Команда go mod init создаёт файл go.mod и включает систему модулей.
- Зависимости добавляются автоматически при импорте и через go get.
- Для обновлений используйте go get -u — как для отдельных библиотек, так и для всех сразу.
- Команда go mod tidy помогает удалить неиспользуемое и навести порядок.
- Внимательное чтение go.mod позволяет контролировать версии и использовать локальные замены (replace).
- Команды go mod tidy, go list и go mod why помогут находить и устранять проблемы в зависимостях.
 
Применение этих действий делает сборку предсказуемой, а проект — устойчивым к внешним изменениям.
 
**Почему это особенно важно в команде:**
- Все разработчики получают одни и те же версии библиотек — меньше сюрпризов.
- Зафиксированные версии снижают риск конфликтов.
- Новым участникам проще влиться в работу, если зависимости описаны и понятны.
- Уязвимости можно быстро закрыть — просто обновив нужную библиотеку.
- Чистота в зависимостях упрощает поддержку и уменьшает «вес» проекта.
 
Поэтому аккуратное управление зависимостями — не просто технический навык, а важная часть культуры разработки. Особенно в командной работе.
