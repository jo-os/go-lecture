## Как отменять горутины и управлять временем выполнения

### Зачем контролировать время работы горутин
Запустить горутину в Go просто: добавляете go перед вызовом функции — и она работает параллельно. Всё выглядит элегантно. Но за этой простотой — серьёзные риски, особенно если горутины живут дольше, чем нужно, или не завершаются вовсе.

В этом разделе — о том, почему нельзя «запустить и забыть» горутину и к чему приводит неконтролируемое выполнение параллельного кода.

**Почему важно останавливать горутины**

Горутина выполняется независимо от основного потока. Если не завершить её вручную, она будет работать до конца жизни процесса.

**Это грозит:**
- утечками памяти (goroutine leak)
- зависаниями в условиях ограниченных ресурсов
- накоплением незавершённых задач
- нестабильным поведением приложения

Горутина — не просто «лёгкая абстракция». Она использует CPU, память, стек и переключения контекста. Сотни или тысячи незавершаемых горутин — это уже проблема.

**Типичная ошибка — забытая горутина**

Пример.

Вы обрабатываете запрос и запускаете горутину для фоновой загрузки данных. Всё работает — пока не случится одно из двух:
- пользователь закрыл соединение
- ответ уже отправлен, но горутина осталась активной

Такая горутина продолжает потреблять ресурсы. При регулярном повторении система перегружается, появляются «призраки» — процессы от уже завершённых запросов.

**Практические последствия в проде**

«Висячие» горутины — частая причина деградации производительности в Go-приложениях. Ниже — реальные кейсы, с которыми сталкивались команды:
- НTTP-сервер обрабатывает 1000 RPS. Каждое обращение запускает горутину, но запросы не отменяются. В минуту — 1000 «лишних» горутин, за час — 60 000. Сервер падает из-за нехватки памяти.
- Фоновый воркер слушает канал. Основной процесс завершается и закрывает канал, но воркер не получает сигнал — и бесконечно грузит CPU.
- Долгие SQL-запросы запускаются в горутине без таймаута. Если БД зависает — горутина ждёт вечно. В проде это мешает масштабированию.

**Что важно спросить себя перед запуском горутины:**
- Когда она должна завершиться?
- Как я узнаю, что она больше не нужна?
- Что будет, если она зависнет?
- Сколько таких горутин может быть одновременно?
- Какие ресурсы она использует?

Ответы помогут писать не просто параллельный, а устойчивый код.

**Как понять, что есть проблема**

Проблемы с горутинами проявляются не сразу. Обратите внимание на такие признаки:
- Количество горутин растёт (runtime.NumGoroutine() увеличивается)
- В профиле — много горутин в статусе waiting
- Память расходуется, но не очищается
- Система тормозит при длительной работе

Подозрительные паттерны:
- Горутину запускают в цикле без условий выхода
- Она ждёт на канале, который может не получить значения
- Нет механизма, чтобы остановить её извне

**Почему недостаточно просто return**

Иногда кажется: «Достаточно выйти из функции — и горутина завершится». Это так, если нет блокировок. Но если в горутине есть ожидание канала, таймера, сетевого соединения — она может зависнуть навсегда без прерывания.

**Чем это грозит в бою**

На проде даже одна забытая горутина может вызвать цепную реакцию. Пример:

- Горутина обрабатывает файл → пользователь отменяет загрузку → горутина продолжает чтение → растёт нагрузка на диск → таких горутин становится всё больше → диск перегружен → падает производительность → ошибки, таймауты, сбой SLA.

Такое случается чаще, чем кажется. Поэтому важно не просто запускать горутины, а уметь их завершать.

**Ключевая идея**

Горутина — не магия, а часть системы. Запуская параллельный процесс, вы отвечаете за его завершение.
Контроль времени выполнения — это не про оптимизацию. Это про надёжность, устойчивость и безопасность приложения.

### Как устроен контекст в Go
Если вы пишете код, который должен быть прерываемым, управляться по времени или передавать данные между слоями (например, логирование, трассировка, авторизация) — без context.Context не обойтись.

Контекст — это единый механизм передачи управляющей информации по стеку вызовов. Он появился в Go 1.7 и стал стандартом де-факто для управления временем жизни операций.

В этом разделе — что такое context.Context, как он работает и зачем он устроен именно так.

**Что такое context.Context**

Это интерфейс, который позволяет:
- Оповещать об отмене (по таймауту, дедлайну или вручную)
- Передавать дедлайн — время завершения
- Хранить значения, доступные по цепочке вызовов

Контекст используется в стандартной библиотеке и сторонних пакетах: сетевой ввод-вывод, работа с БД, брокеры сообщений и др.

Сигнатура интерфейса:
```go
type Context interface {
Deadline() (deadline time.Time, ok bool)
Done() <-chan struct{}
Err() error
Value(key any) any
}
```
**Основные методы**

- Deadline() - Возвращает момент, до которого контекст действителен. Если не задан — ok == false.

Пример: deadline, ok := ctx.Deadline()

- Done() - Канал, который закрывается при завершении контекста — вручную или по таймауту.

Не передаёт значения, только закрывается — удобно использовать в select.

- Err() - Возвращает причину завершения:
  - nil — контекст активен
  - context.Canceled — отмена вручную
  - context.DeadlineExceeded — превышено время

- Value()
Позволяет передавать значения по дереву вызовов.
Например, userID или traceID.
Важно: значения должны быть неизменяемыми и использоваться строго по назначению — контекст не хранилище.

**Контексты образуют дерево**

Каждый контекст связан с родителем. Это значит:
- Отмена родителя завершает все дочерние контексты
- Таймауты и дедлайны наследуются

Пример иерархии:
```
rootCtx
├── childCtx1
│ └── grandChildCtx1
└── childCtx2
└── grandChildCtx2
```
Если отменён rootCtx — завершатся все потомки.

**Контексты создаются через функции**

Вы не создаёте context.Context напрямую — используете конструкторы из пакета context:
```go
ctx := context.Background()
ctx2, cancel := context.WithCancel(ctx)
ctx3, cancel := context.WithTimeout(ctx, 2*time.Second)
ctx4 := context.WithValue(ctx, key, value)
```
**Семантика основных функций:**
- Background() — корневой контекст, не завершается сам
- TODO() — временный заглушка
- WithCancel() — отмена вручную
- WithTimeout() — завершение по таймауту
- WithDeadline() — завершение в указанный момент
- WithValue() — добавление значений

**Где используется context.Context**

Почти везде:
- В http.Request есть метод _Context()_ — по нему можно определить, жив ли запрос
- В database/sql — db.QueryContext(ctx, ...)
- В gRPC — все хендлеры принимают context.Context как первый аргумент
- В AWS SDK, Kafka, Redis и др. — используется для отмены, дедлайнов и логирования

**Почему это важно**

Контекст — единый способ управлять временем жизни операций и передавать сквозные данные.
До него каждая библиотека решала это по-своему: флаги, таймеры, каналы, костыли. Это мешало взаимодействию между компонентами.

Теперь, если функция принимает ctx context.Context, вы сразу понимаете:
- её можно прервать
- можно задать дедлайн
- можно передать сквозные данные

context.Context — универсальный стандарт, который упростил взаимодействие в экосистеме Go.

**Как правильно использовать контекст**
- Создавайте снаружи, передавайте внутрь. Вложенные функции не должны создавать собственные корневые контексты.
- Не сохраняйте между вызовами. Контекст живёт только в рамках конкретной операции.
- Не модифицируйте контекст напрямую. Он неизменяем. Любое изменение — создание нового контекста на основе предыдущего.
- Не используйте как контейнер аргументов. WithValue — не замена параметрам функции. Используйте его только для сквозных данных (логгер, trace-id и т. п.).

**Что запомнить:**
- сontext.Context управляет временем жизни операций
- Передаёт дедлайны, отмену и значения
- Контексты образуют дерево: отмена родителя завершает потомков
- Используется во всех современных библиотеках
- Создавайте контекст на верхнем уровне и передавайте вниз
- WithValue — только для сквозных данных, не для бизнес-логики

### Примеры отмены горутин
Понимая, зачем управлять временем жизни горутин и как устроен context.Context, перейдём к практике. В этом разделе — как отменять горутины вручную с помощью context.WithCancel.

**Когда нужна отмена горутины**

Горутина должна завершаться, если:
- основной процесс или запрос завершён
- задача стала неактуальной
- пользователь отменил действие
- возникла ошибка на верхнем уровне

Если не обеспечить выход, горутина продолжит потреблять ресурсы.

**Отмена через context.WithCancel**

Функция context.WithCancel(parent) возвращает новый контекст и функцию отмены:

ctx, cancel := context.WithCancel(parent)
- ctx — новый контекст
- cancel — функция, которую нужно вызвать для отмены

После вызова cancel() канал ctx.Done() закрывается, и все слушающие это замечают.

**Базовый шаблон**
```go
func main() {
ctx, cancel := context.WithCancel(context.Background())
defer cancel()
go worker(ctx)
time.Sleep(2 * time.Second)
cancel()
time.Sleep(1 * time.Second)
}
func worker(ctx context.Context) {
for {
select {
case <-ctx.Done():
fmt.Println("Горутина завершена")
return
default:
fmt.Println("Работаю...")
time.Sleep(500 * time.Millisecond)
}
}
}
```
**Пояснение:**
- Создаём контекст с cancel()
- Запускаем горутину с этим ctx
- Внутри горутина слушает ctx.Done()
- После cancel() — завершение

**Почему важно вызывать cancel()**

Если вы не вызовете cancel() — ресурсы, связанные с контекстом, не освободятся.

**Поэтому всегда:**
- вызывайте cancel() вручную после завершения операции
- или используйте defer cancel() сразу после создания контекста

**Пример с отменой по действию пользователя**
```go
func main() {
ctx, cancel := context.WithCancel(context.Background())
defer cancel()
go longOperation(ctx)
fmt.Println("Нажмите Enter для отмены")
fmt.Scanln()
cancel()
time.Sleep(1 * time.Second)
}
func longOperation(ctx context.Context) {
for i := 0; ; i++ {
select {
case <-ctx.Done():
fmt.Println("Операция отменена пользователем")
return
default:
fmt.Printf("Шаг %d\n", i)
time.Sleep(1 * time.Second)
}
}
}
```
**Пример вывода:**
```
Нажмите Enter для отмены
Шаг 0
Шаг 1
Шаг 2
<нажат Enter>
```
Операция отменена пользователем

Такой шаблон позволяет реагировать на внешние события — ввод пользователя, сигналы ОС, команды интерфейса и т. п.

**Отмена нескольких горутин**

Вы можете использовать один контекст для управления сразу несколькими горутинами. Все, кто слушает ctx.Done(), получат сигнал одновременно.
```go
func main() {
ctx, cancel := context.WithCancel(context.Background())
defer cancel()
for i := 0; i < 3; i++ {
go worker(ctx, i)
}
time.Sleep(3 * time.Second)
fmt.Println("Останавливаем все горутины...")
cancel()
time.Sleep(1 * time.Second)
}
func worker(ctx context.Context, id int) {
for {
select {
case <-ctx.Done():
fmt.Printf("Горутина #%d завершена\n", id)
return
default:
fmt.Printf("Горутина #%d работает\n", id)
time.Sleep(500 * time.Millisecond)
}
}
}
```
**Пример вывода:**
```
Горутина #0 работает
Горутина #1 работает
Горутина #2 работает
...
Останавливаем все горутины...
Горутина #0 завершена
Горутина #1 завершена
Горутина #2 завершена  
```
**Универсальный шаблон отменяемой горутины**
```go
func DoSomething(ctx context.Context) error {
for {
select {
case <-ctx.Done():
return ctx.Err()
case data := <-someChannel:
process(data)
}
}
}
```
**Такой шаблон:**
- слушает ctx.Done()
- обрабатывает данные
- корректно завершает работу с ошибкой
**Передавайте контекст по стеку**

Хорошая практика — не создавать новый контекст внутри каждой функции, а передавать полученный:
```go
func A(ctx context.Context) {
B(ctx)
}
func B(ctx context.Context) {
go C(ctx)
}
```
Это сохраняет единый жизненный цикл операции.

**Что должна уметь отменяемая горутина**
- слушать ctx.Done()
- освобождать ресурсы (соединения, файлы и пр.)
- завершаться быстро и безопасно
- при необходимости — проверять ctx.Err() для понимания причины

**Что делать не стоит**
- Не проверяйте ctx.Err() в отдельном цикле — используйте select
- Не игнорируйте cancel() — даже если отмена не требуется, это освобождает ресурсы
- Не создавайте отдельный контекст на каждую горутину без необходимости

**Вывод**

context.WithCancel позволяет:
- централизованно завершать процессы
- безопасно очищать ресурсы
- избегать утечек

Это один из базовых механизмов при работе с фоновыми задачами, сетевыми запросами и асинхронной логикой в Go.

**Как работают таймауты и дедлайны**

При работе с горутинами, сетевыми запросами, БД и другими долгими операциями важно ограничивать время выполнения.

В Go для этого есть два встроенных механизма:
- сontext.WithTimeout — завершение через заданное время
- сontext.WithDeadline — завершение в конкретный момент времени

В чём разница
- Таймаут — относительное время: «через 2 секунды отменить»
- Дедлайн — абсолютное время: «не позже 15:04:05»

Оба создают контекст, который автоматически отменяется при наступлении срока.

Пример: WithTimeout
```go
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

Через 3 секунды:

• ctx.Done() закроется
• ctx.Err() вернёт context.DeadlineExceeded
```
Пример: WithDeadline
```go
deadline := time.Now().Add(3 * time.Second)
ctx, cancel := context.WithDeadline(context.Background(), deadline)
defer cancel()
```
Работает так же, но позволяет задать конкретное время (например, из конфигурации или заголовка запроса).

**Шаблон отмены по таймауту**
```go
func main() {
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()
doWork(ctx)
}
func doWork(ctx context.Context) {
select {
case <-time.After(3 * time.Second):
fmt.Println("Работа завершена")
case <-ctx.Done():
fmt.Println("Работа отменена:", ctx.Err())
}
}
```
Что произойдёт:
- через 2 секунды сработает таймаут
- контекст будет отменён
- сtx.Done() закроется
- выведется сообщение об отмене

**Зачем это нужно**
Без таймаута операция может зависнуть надолго — особенно при работе с внешними сервисами.
С таймаутом вы ограничиваете время ожидания и сохраняете управляемость.

**Где применять**

Таймауты нужны, если:
- операция завязана на отклик пользователю
- есть внешние зависимости (БД, API)
- задача может «повиснуть»
- превышение времени критично для ресурсов

Типичные случаи:
- запросы к БД
- внешние HTTP-запросы
- тяжёлые расчёты
- реакции на действия пользователя

Что происходит под капотом
- Go запускает таймер
- По срабатыванию:
  - закрывается ctx.Done()
  - сtx.Err() → context.DeadlineExceeded
  - освобождаются ресурсы

Если вызвать cancel() раньше — отмена произойдёт немедленно.

**Почему важно вызывать cancel()**

Если вы создаёте контекст с WithTimeout или WithDeadline, вы обязаны вызвать cancel(), даже если не планируете ручную отмену. Почему это важно:
- Таймер и связанные ресурсы (в том числе внутренняя горутина) не освобождаются до истечения срока
- Без вызова cancel() возникает утечка: операция уже завершена или не нужна, но система продолжает ожидание

Поэтому всегда вызывайте cancel() — чаще всего через defer cancel() сразу после создания контекста.

**Пример: функция с таймаутом**
```go
func workWithTimeout() {
ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
defer cancel()
done := make(chan string)
go func() {
time.Sleep(2 * time.Second)
done <- "готово"
}()
select {
case result := <-done:
fmt.Println("Результат:", result)
case <-ctx.Done():
fmt.Println("Превышен лимит времени:", ctx.Err())
}
}
```
**Что произойдёт:**
- горутина работает 2 секунды
- контекст живёт только 1 секунду
- сtx.Done() срабатывает первым
- результат: Превышен лимит времени: context deadline exceeded

Таймауты обеспечивают устойчивость.

Они делают поведение системы контролируемым:
- операции не зависают
- задачи завершаются в пределах ожидаемого времени
- можно корректно обработать превышение времени

**Что выбрать — таймаут или дедлайн?**

Выбор зависит от контекста:

|Когда использовать|Что выбрать|
|------------------|-----------|
|Известна длительность|WithTimeout|
|Задан конкретный момент|WithDeadline|

Функционально оба работают одинаково — отличается только способ задания времени.

**Объединение таймаута и ручной отмены**

Можно использовать оба механизма:
```go
parent := context.Background()
ctx, cancel := context.WithTimeout(parent, 5*time.Second)
defer cancel()
go doWork(ctx)
// Позже — ручная отмена:
cancel()
```
Контекст завершится:
- через 5 секунд по таймауту
- или немедленно при ручной отмене

Рекомендации:
- Всегда вызывайте cancel() для контекстов с таймером
- Не передавайте context.Background() вглубь — используйте его только в main() или init()
- Не задавайте слишком короткие таймауты без анализа — возможны ложные ошибки
- Не игнорируйте ctx.Err() — он указывает причину завершения

**Сравнение по функциям**
|Элемент|Назначение|
|-------|----------|
|WithTimeout|Завершает контекст через заданное время|
|WithDeadline|Завершает в заданный момент времени|
|ctx.Done()|Закрывается при отмене|
|ctx.Err()|Возвращает context.Canceled или DeadlineExceeded|
|cancel()|Завершает контекст вручную|

**Вывод**

Контексты с таймерами — основа надёжного асинхронного кода в Go. Это не «дополнительная защита», а часть архитектурного подхода, необходимого для масштабируемых и управляемых систем.

### Как корректно завершать горутины с каналами
При работе с каналами горутина может «зависнуть» в ожидании данных. Если она не узнает, что данных больше не будет, — это приводит к утечке ресурсов.

**Проблема: «вечное» ожидание**

Пример:
```go
func worker(jobs <-chan int) {
for job := range jobs {
fmt.Println("Обработка", job)
}
}
```
Если jobs никогда не закроется, горутина останется в ожидании навсегда — даже когда работа фактически завершена.

**Как завершать горутину, слушающую канал**

Варианты:
- Закрыть канал (и читать через for range)
- Передать сигнальное значение ("quit", -1, nil)
- Использовать ctx.Done() в select

На практике лучше всего работает комбинация канала + контекста.

**Универсальный шаблон с ctx.Done() и каналом**
```go
func worker(ctx context.Context, jobs <-chan int) {
for {
select {
case <-ctx.Done():
fmt.Println("Отмена горутины")
return
case job, ok := <-jobs:
if !ok {
fmt.Println("Канал закрыт")
return
}
fmt.Println("Обработка", job)
}
}
}
```
Этот шаблон:
- предотвращает зависания
- корректно реагирует на отмену
- безопасно завершает горутину при закрытии канала

Почему нельзя полагаться только на закрытие канала:
- В Go канал может закрыть только один отправитель — при множественных продюсерах это риск
- При нескольких каналах (данные, ошибки и т. п.) закрытие одного не даёт сигнал другим
- При внешнем управлении (например, HTTP-запрос) нужен механизм отмены «сверху»

Пример: продюсер и воркер с отменой
```go
func main() {
ctx, cancel := context.WithCancel(context.Background())
jobs := make(chan int)
go producer(ctx, jobs)
go worker(ctx, jobs)
time.Sleep(2 * time.Second)
cancel()
time.Sleep(1 * time.Second)
}
func producer(ctx context.Context, jobs chan<- int) {
defer close(jobs)
for i := 0; ; i++ {
select {
case <-ctx.Done():
fmt.Println("Продюсер завершён")
return
case jobs <- i:
fmt.Println("Отправлено:", i)
time.Sleep(300 * time.Millisecond)
}
}
}
func worker(ctx context.Context, jobs <-chan int) {
for {
select {
case <-ctx.Done():
fmt.Println("Воркер завершён")
return
case job, ok := <-jobs:
if !ok {
fmt.Println("Канал закрыт — воркер завершён")
return
}
fmt.Println("Обработка", job)
time.Sleep(500 * time.Millisecond)
}
}
}
```
**Что здесь важно:**
- и продюсер, и воркер слушают ctx.Done()
- продюсер закрывает канал, когда отменён
- воркер завершает работу при отмене или закрытии канала

Зачем использовать ctx.Done()
- Можно слушать вместе с другими каналами
- Позволяет управлять из внешнего слоя
- Закрывается автоматически при отмене, таймауте или дедлайне

**Обработка нескольких каналов**

Когда горутина работает с несколькими каналами (например, data, errors, done), добавляйте ctx.Done() в select, чтобы можно было остановить выполнение централизованно:
```go
select {
case <-ctx.Done():
log.Println("Операция отменена")
return
case msg := <-data:
process(msg)
case err := <-errors:
handle(err)
}
```
Такой подход:
- предотвращает зависания
- позволяет реагировать на внешнюю отмену
- безопасно завершает горутину

**Как завершать горутину, которая пишет в канал**

Писать в канал нужно с осторожностью. Возможные риски:
- если получателя нет — горутина блокируется
- если канал закрыт — возникает panic
- если не реагировать на отмену — будет утечка

**Безопасный способ отправки:**
```go
func safeSend(ctx context.Context, ch chan<- int, value int) bool {
select {
case <-ctx.Done():
return false
case ch <- value:
return true
}
}

Внутри продюсера:
if !safeSend(ctx, ch, i) {
fmt.Println("Прерывание отправки")
return
}
```
**Когда закрывать канал**

**Правило:** канал закрывает только тот, кто его создал, и только если он — единственный отправитель.

**Если в канал пишут несколько горутин:**
- канал лучше не закрывать — сборщик мусора справится
- используйте другие способы окончания: WaitGroup, ctx, done-канал

**Чего не стоит делать**
- писать в закрытый канал — вызовет panic
- не проверять ok при чтении из канала
- не слушать ctx.Done()
- полагаться только на close(ch)
- терять владение каналом (глобальные переменные и т. п.)

Универсальный шаблон чтения с отменой
```go
func readLoop(ctx context.Context, ch <-chan string) {
for {
select {
case <-ctx.Done():
fmt.Println("Чтение остановлено:", ctx.Err())
return
case msg, ok := <-ch:
if !ok {
fmt.Println("Канал закрыт")
return
}
fmt.Println("Получено:", msg)
}
}
}
```
Итого: как завершать горутины с каналами

|Подход|Когда применять|
|------|---------------|
|close(ch) + for range|если вы полностью контролируете канал|
|ctx.Done() в select|всегда, если горутина должна быть отменяемой|
|Проверка ok при чтении|чтобы выйти при закрытии канала
|Безопасная отправка|если получатель может отмениться
|Комбинированные шаблоны|для надёжной интеграции компонентов

### Отмена в HTTP-серверах
Go запускает отдельную горутину на каждый входящий HTTP-запрос. Если клиент закрыл соединение, а обработка всё ещё продолжается — сервер зря расходует ресурсы. Важно уметь реагировать на отмену запроса и корректно завершать связанные операции.

**Когда нужна отмена запроса**

Задумайтесь об отмене, если:
- внутри http.HandlerFunc выполняются долгие вычисления
- идёт обращение к БД или внешним API
- используются горутины, таймеры, каналы
- нужно как можно раньше освободить ресурсы

Пример: клиент нажал «Стоп» в браузере, но сервер продолжает выполнять тяжёлый SQL-запрос — результат уже не нужен, а ресурсы расходуются.

**Контекст запроса в Go**

Каждому HTTP-запросу автоматически присваивается context.Context. Получить его можно так:
```go
func handler(w http.ResponseWriter, r *http.Request) {
ctx := r.Context()
// используйте ctx
}
```
**Контекст отменяется автоматически, если:**
- клиент закрыл соединение
- сработал таймаут http.Server
- сервер завершил работу

Используйте ctx.Done() для корректного завершения обработки.

**Пример 1: обработка с прерыванием**
```go
func handler(w http.ResponseWriter, r *http.Request) {
ctx := r.Context()
select {
case <-time.After(5 * time.Second):
fmt.Fprintln(w, "Готово")
case <-ctx.Done():
log.Println("Запрос отменён:", ctx.Err())
}
}
```
Если клиент ушёл до завершения таймера, срабатывает ctx.Done() — ответ не отправляется, и ошибка broken pipe не возникает.

**Почему нельзя писать в http.ResponseWriter после отмены**

Если клиент отключился, а вы продолжаете писать в w, произойдёт ошибка записи (например, broken pipe). Поэтому важно завершить обработку до попытки ответа.

**Пример 2: отменяемый запрос к БД**
```go
func handler(w http.ResponseWriter, r *http.Request) {
ctx := r.Context()
db := getDB()
rows, err := db.QueryContext(ctx, "SELECT pg_sleep(10)")
if err != nil {
log.Println("Ошибка запроса:", err)
http.Error(w, "Ошибка", http.StatusInternalServerError)
return
}
defer rows.Close()
fmt.Fprintln(w, "Успех")
}
```
Если клиент отключился, ctx отменится, и QueryContext остановит SQL-запрос. Это работает только с методами, принимающими context.Context.

**Пример 3: фоновая горутина внутри обработчика**
```go
func handler(w http.ResponseWriter, r *http.Request) {
ctx := r.Context()
go func() {
select {
case <-time.After(10 * time.Second):
log.Println("Готово")
case <-ctx.Done():
log.Println("Горутина отменена:", ctx.Err())
}
}()
fmt.Fprintln(w, "Ответ отправлен")
}
```
Если не слушать ctx.Done(), горутина продолжит работу даже после завершения запроса — это утечка.

**Обработка отмены внутри своей логики**

Если вы пишете функцию, которую можно отменить — она должна принимать ctx context.Context:
```go
func doWork(ctx context.Context) error {
for i := 0; i < 10; i++ {
select {
case <-ctx.Done():
return ctx.Err()
default:
time.Sleep(500 * time.Millisecond)
}
}
return nil
}
```
Применение в HTTP-обработчике:
```go
func handler(w http.ResponseWriter, r *http.Request) {
err := doWork(r.Context())
if err != nil {
log.Println("Прервано:", err)
http.Error(w, "Отмена", http.StatusRequestTimeout)
return
}
fmt.Fprintln(w, "Готово")
}
```
**Таймаут на уровне сервера**

Глобально:
```go
srv := &http.Server{
Addr: ":8080",
Handler: myHandler,
ReadTimeout: 5 * time.Second,
WriteTimeout: 10 * time.Second,
}
```
Локально (на один обработчик):
```go
func withTimeout(h http.HandlerFunc) http.HandlerFunc {
return func(w http.ResponseWriter, r *http.Request) {
ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
defer cancel()
r = r.WithContext(ctx)
h(w, r)
}
}
```
**Где отмена критична**

Контекст нужен, если:
- вы обращаетесь к БД: QueryContext, ExecContext
- делаете HTTP-запросы: http.NewRequestWithContext
- используете каналы, select, таймеры
- запускаете фоновые горутины

Главное правило — передавайте ctx везде и проверяйте ctx.Done() внутри долгих операций.

Полезные паттерны
- передавайте ctx в собственные функции
- оборачивайте сторонние вызовы в select { case <-ctx.Done() ... }
- не запускайте горутины без ctx.Done()
- используйте Request.WithContext(...), если нужно задать таймаут

Резюме: что вы должны уметь
|Навык|Зачем это нужно|
|-----|---------------|
|Использовать r.Context()|Контекст автоматически привязан к запросу|
|Реагировать на <-ctx.Done()|Чтобы не работать зря|
|Передавать ctx в бизнес-логику
Использовать методы ...Context()|Для контроля времени выполнения|Использовать методы ...Context()|
|Для БД, HTTP, сторонних библиотек|Не писать в ResponseWriter после отмены|
|Во избежание ошибок записи|Проверять ctx.Err()|Для понимания причины отмены|

### Типичные ошибки и как их избежать
1. Не вызвали cancel()
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
// Ошибка: cancel не вызван
```
Правильно:
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
```
2. Запустили горутину без ctx.Done()
```go
go func() {
time.Sleep(10 * time.Second)
doSomething()
}()
```
Правильно:
```go
go func(ctx context.Context) {
select {
case <-time.After(10 * time.Second):
doSomething()
case <-ctx.Done():
log.Println("Горутина отменена:", ctx.Err())
}
}(ctx)
```
3. Не проверили ctx.Err()
```go
case <-ctx.Done():
// ошибка не обработана
```
Правильно:
```go
case <-ctx.Done():
switch ctx.Err() {
case context.Canceled:
log.Println("Отменено вручную")
case context.DeadlineExceeded:
log.Println("Истёк таймаут")
}
```
4. Не передали ctx в вызов
```go
result := db.Query("SELECT ...") // контекст игнорируется
```
Правильно:
```go
result := db.QueryContext(ctx, "SELECT ...")
```
5. select без default в долгом цикле
```go
for {
select {
case msg := <-ch:
handle(msg)
case <-ctx.Done():
return
}
}
```
Если ch заблокирован, отмена может не сработать. Убедитесь, что логика контролируется внешними сигналами.

6. Преждевременная отмена родителя
```go
parent, parentCancel := context.WithCancel(context.Background())
child, childCancel := context.WithCancel(parent)
parentCancel() // отменяет и child
```
Убедитесь, что родительский контекст не завершается раньше времени, если в нём живут активные дочерние процессы.

7. Использование context.Background() внутри функций
```go
func fetchData() {
ctx := context.Background() // ошибка
}
```
Правильно:
```go
func fetchData(ctx context.Context) {
// ...
}
```
Рекомендации
|Рекомендация|Причина|
|------------|-------|
|Всегда вызывайте cancel()|Освобождает ресурсы|
|Передавайте ctx во все функции|Поддерживает иерархию управления|
|Используйте методы ...Context()|Поддержка отмены в сторонних вызовах|
|Слушайте <-ctx.Done() в горутинах|Предотвращает утечки|
|Проверяйте ctx.Err()|Для логики и отладки|
|Не используйте context.Background() внутри|Обрезает цепочку отмен|
|Не пишите в ResponseWriter после отмены|Может вызвать ошибку|
|Документируйте поведение при отмене|Повышает читаемость и предсказуемость|

### Вывод
context.Context — мощный инструмент, но при неправильном применении он становится источником сложных багов. Ошибки с контекстом могут привести к:
- утечкам памяти
- бесконтрольному росту горутин
- избыточной нагрузке на БД
- таймаутам и сбоям в проде

**Хорошая функция:**
- принимает ctx context.Context
- слушает ctx.Done()
- корректно завершает горутины
- освобождает ресурсы и вызывает cancel()

Такой подход делает ваш код надёжным, предсказуемым и готовым к работе в условиях высокой нагрузки.
