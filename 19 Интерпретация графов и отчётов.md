## Интерпретация графов и отчётов

Вы запускаете программу, и она работает медленно. Но где именно «тормозит»? В логах ошибок нет, тесты проходят, а ощущение, что ресурсы уходят впустую.

В таких случаях помогают инструменты профилировки. Они показывают, что происходит внутри программы: какие функции загружают CPU, где расходуется память, когда и почему горутины блокируются.

### 1. Профилировка как инструмент понимания
В Go есть два встроенных помощника для такой диагностики — pprof и trace. Они работают по-разному и нужны для разных задач.

**pprof — агрегированная картина**

Даёт сводную статистику по CPU, памяти, горутинам и блокировкам.

Помогает:
- увидеть «горячие точки» кода;
- понять, где расходуется больше всего памяти;
- отследить частые вызовы функций.

Типы профилей:
- CPU — распределение процессорного времени;
- Heap — использование памяти;
- Goroutine — состояние и количество горутин;
- Block / Mutex — задержки на блокировках.

Форматы отображения: текст, таблицы, граф вызовов, flamegraph.

**trace — поведение во времени**

Показывает выполнение программы по временной шкале: создание горутин, переключения, блокировки, системные вызовы. Это инструмент для точечной диагностики.

С его помощью можно:
- понять, как задачи распределяются по потокам;
- измерить время ожидания и выполнения;
- увидеть моменты, когда синхронизация тормозит выполнение.

Визуализация: timeline с детализацией до каждого события.



**Когда что использовать**
| ... |pprof|trace|
|-----|-----|-----|
|Что показывает|Сводку по ресурсам|Хронологию событий|
|Детализация|Функции, пакеты, блоки кода|Переключения горутин, синхронизацию|
|Влияние на код|Умеренное|Высокое — применять выборочно|
|Основная задача|Найти «узкие места»|Разобрать поведение во времени|

Оптимальный подход: начинаем с pprof для общей картины, затем подключаем trace для глубокой диагностики.

**Как читать flamegraph (pprof)**

Flamegraph — это диаграмма, где:
- прямоугольник = функция;
- ширина = сколько времени она заняла (вместе с вызовами потомков);
- высота = стек вызовов;
- цвет — просто различие, смысла не несёт.

Как работать:
- Читаем снизу вверх — это стек вызовов.
- Смотрим на ширину: чем шире, тем больше времени.
- Широкие «одиночные» функции — кандидаты в узкие места.
- Глубокое, но узкое дерево — обычно не проблема.

Пример:
```
main.main
└── handler
└── parseRequest
└── decodeJSON
└── strings.Split
```

Если decodeJSON шире parseRequest — основное время уходит туда. Если вся ширина в strings.Split, значит, тормоз там.

**Как читать timeline (trace)**

Timeline — временная шкала активности горутин.
- X — время работы программы
- Y — горутины/потоки
- цветные блоки — выполнение, ожидание, блокировки

Чаще всего:
- синий — выполнение;
- жёлтый — ожидание блокировки;
- красный — системные вызовы;
- серый — бездействие.

Что искать:
- долгие ожидания одной горутины;
- короткие, частые переключения;
- зависания (нет активности);
- пики нагрузки (все полосы активны).

Важные метрики в trace
- Network blocking — сетевые задержки;
- Synchronization blocking — ожидание мьютексов/каналов;
- Goroutine analysis — время жизни и количество горутин;
- Processor utilization — загрузка ядер.

Flamegraph и timeline — не «красивая картинка», а инструмент анализа:
- переводят субъективное «медленно» в факты;
- показывают распределение времени по коду;
- помогают локализовать перегрузку, блокировки и неэффективность.

Работайте с ними как с картой: от общей картины к точкам, где прячется проблема.

### 2. pprof — пример
Рассмотрим, как запустить CPU-профилирование Go-программы и что можно ожидать в отчёте.

**Пример: неэффективная конкуренция**

Код обрабатывает 10 000 «запросов» в отдельных горутинах. Каждая горутина спит 10 мс.
```go
package main
import (
"log"
"os"
"runtime/pprof"
"sync"
"time"
)
func handleRequest(id int) {
time.Sleep(10 * time.Millisecond)
}
func main() {
f, err := os.Create("cpu.prof")
if err != nil {
log.Fatal(err)
}
defer f.Close()
if err := pprof.StartCPUProfile(f); err != nil {
log.Fatal(err)
}
defer pprof.StopCPUProfile()
var wg sync.WaitGroup
wg.Add(10_000)
for i := 0; i < 10_000; i++ {
go func(id int) {
defer wg.Done()
handleRequest(id)
}(i)
}
wg.Wait()
time.Sleep(100 * time.Millisecond)
}
```
Что делает код
- Создаёт файл cpu.prof — туда пишется бинарный профиль CPU.
- Запускает pprof.StartCPUProfile — фиксирует активное время выполнения функций.
- Запускает 10 000 горутин и ждёт их завершения через WaitGroup.
- Останавливает профилирование (StopCPUProfile) и ждёт 100 мс для записи данных.

На что обратить внимание
- Профиль CPU учитывает только активное выполнение, а не ожидание (Sleep, блокировки).
- 10 000 горутин на 4–8 ядер → высокая конкуренция, частые переключения, накладные расходы планировщика.
- В отчёте будут заметны функции runtime.* — это сигнал, что часть ресурсов уходит на управление горутинами, а не на полезную работу.

Мини-инструкция для подключения pprof
```
Файл профиля
f, _ := os.Create("cpu.prof")
```
- Открывать: go tool pprof, go tool pprof -http, speedscope, flamegraph.
- Старт - pprof.StartCPUProfile(f)
- Стоп - pprof.StopCPUProfile()
- Пауза перед выходом — даёт завершить буферизацию.

Почему CPU-время низкое

В handleRequest единственная операция — time.Sleep(10 * time.Millisecond). Все горутины ждут, не выполняя полезной работы.

Это имитирует I/O-нагрузку, при которой:
- pprof фиксирует только активное выполнение, поэтому CPU-время окажется низким;
- цикл «создание → ожидание → завершение» тратит ресурсы планировщика и системных вызовов;
- в профиле доминируют функции runtime/syscall, а пользовательского кода мало.

Вывод: pprof измеряет работу процессора, а не время жизни горутины.

**Быстрый старт с pprof**
- Соберите и запустите: - go build -o app && ./app
- Получите профиль CPU (через код или /debug/pprof/profile?seconds=10).
- Откройте в консоли: go tool pprof cpu.prof - Команды: top, list <func>, web, svg, quit.
- Или запустите веб-интерфейс: - go tool pprof -http=:8088 cpu.prof

**Что дальше**

Передайте файл профиля — по нему определим горячие функции, оценим использование параллелизма и contention, найдём узкие места и объясним, почему производительность падает даже при большом числе горутин.

Итоги
- Профилировать нужно под реальной нагрузкой.
- CPU-профиль фиксирует выполнение, а не ожидание (sleep, I/O).
- Много горутин ≠ высокая пропускная способность.
- Дальше — анализ профиля (top, list) и просмотр flamegraph.

**Разбор профиля**
- runtime.cgocall — 80 мс (40%)
  - Появляется из-за системных вызовов в time.Sleep.
  - На Windows sleep обращается к API через cgo, тратя CPU на переход Go→C. Вывод: даже «пустой» sleep нагружает CPU системными вызовами.
- runtime.stdcall* — 50 мс (25%)
  - Внутренности syscall.SyscallN (разные варианты по числу аргументов).
  - Подтверждает: нагрузка идёт на нативные системные функции, а не на Go-код.
- sync.WaitGroup — 30 мс (15%)
  - Основное время в атомарных операциях Add и Done.
  - Высокая конкуренция 10 000 горутин делает WG узким местом при масштабировании.
- runtime.stack* — 60 мс (30%)
  - Выделение и перераспределение стеков для новых горутин.
  - Массовое создание короткоживущих горутин перегружает планировщик.
- runtime.(*timers).* — 30 мс (15%)
  - Управление кучей таймеров для всех time.Sleep.
  - Десятки тысяч одновременных таймеров → деградация производительности.

**Что отсутствует:**
- handleRequest не в топе — в нём только sleep, «полезной» работы нет.
- main.main и main.main.func1 — всего по 10% времени. В нормальной ситуации они должны быть лидерами.

**Итоги: где узкое место**

Топ затрат:
- runtime.cgocall — 40%: системные вызовы (в т.ч. Sleep) через C.
- runtime.stdcall* — 25%: обёртки для системных API.
- sync.WaitGroup — 15%: атомарные операции счётчика.
- runtime.stackpoolalloc — 15%: выделение стеков под горутины.
- runtime.(*timers) — 15%: управление таймерами Sleep.

**Вывод:**

10 000 горутин с коротким Sleep не грузят CPU, но перегружают планировщик, стековую аллокацию и таймеры. Полезная работа минимальна, потери — в накладных расходах Go-рантайма и ОС.

**Заключение:**

Это классический антипаттерн: массовый запуск тысяч горутин с ожиданием создаёт иллюзию параллелизма, но фактически снижает производительность. pprof позволяет доказать это фактами, а не предположениями.

### 3. trace — зачем, если есть pprof
pprof отлично отвечает на вопросы:
- что занимало CPU;
- какие функции самые «тяжёлые»;
- где сосредоточено потребление ресурсов.

Но бывают ситуации, когда CPU не загружен, а программа всё равно «тормозит»: горутины простаивают, отклик скачет, параллелизма почти нет. Здесь нужен trace.

Что даёт trace:
- детальную временную картину работы программы;
- события: запуск, блокировка, пробуждение горутин;
- распределение задач по потокам;
- простои и очереди на выполнение;
- влияние GC и системных событий.

Пример: неочевидная задержка
```go
package main
import (
"log"
"os"
"runtime/trace"
"sync"
"time"
)
func handleRequest(id int) {
time.Sleep(10 * time.Millisecond) // имитация работы
}
func main() {
f, _ := os.Create("trace.out")
defer f.Close()
trace.Start(f)
defer trace.Stop()
var wg sync.WaitGroup
wg.Add(10_000)
for i := 0; i < 10_000; i++ {
go func(id int) {
defer wg.Done()
handleRequest(id)
}(i)
}
wg.Wait()
time.Sleep(100 * time.Millisecond) // для записи данных
}
```
Что делает программа:
- записывает события выполнения в trace.out;
- запускает 10 000 горутин, каждая спит 10 мс;
- ждёт их завершения.

С виду — корректная параллельная обработка. В trace же видно: значительная часть времени уходит на переключение контекста и ожидание ресурсов.

**Что искать в trace**
- Задержки между готовностью и стартом горутины.
- Массовые блокировки и простои CPU.
- Перегрузку планировщика при большом числе задач.
- Очереди ожидания выполнения.
- Паузы, вызванные GC или системными событиями.

Как запустить и просмотреть trace
- Добавьте в код:
```go
import "runtime/trace"
trace.Start(file)
...
trace.Stop()
```
- Запустите программу — появится trace.out.
- Откройте в браузере: go tool trace trace.out
- Веб-интерфейс покажет временную шкалу, статистику и отчёты по событиям.

**Интерфейс go tool trace**

При открытии trace в браузере доступны основные разделы:
- Event timelines
  - View trace by proc — по логическим процессорам (P).
  - View trace by thread — по потокам ОС. Здесь видна временная шкала: работа горутин, блокировки, ожидания.
- Goroutine analysis
  - Группировка по функциям. Позволяет увидеть, сколько горутин работают, ждут или заблокированы (например, все запущенные через handleRequest).
- Profiles - Четыре вида блокирующих профилей:
  - Network blocking
  - Synchronization blocking
  - Syscall
  - Scheduler latency
  - Показывают, почему горутины простаивали (данные из trace в формате pprof).

**Вкладка Execution Trace (View trace by proc)**

Главный инструмент: временная диаграмма исполнения.
- Ось X — время работы программы.
- Ось Y — логические процессоры.
- Цветные сегменты — выполнение, ожидание, блокировка.

**Как читать Execution Trace**

Цвета:
- Синий — выполнение горутины
- Жёлтый — готова, но ждёт CPU.
- Серый — блокировка (WaitGroup, каналы, мьютексы).
- Фиолетовый — работа GC.
- Красный — системные вызовы (sleep, ожидание I/O).

Ширина блока = длительность события.

Горизонтальные линии — потоки ОС (M), на которых планировщик Go (P) исполняет горутины. Пробелы на линии — простаивание или ожидание.

**Зачем это нужно**

trace показывает не только что выполнялось, но и когда, как долго и почему не выполнялось.
- Видно, как горутины взаимодействуют, блокируют друг друга, ждут CPU.
- Можно обнаружить перегрузку планировщика (много горутин «готовы, но ждут»).
- Прерывания из-за GC или системных вызовов сразу заметны на временной шкале.

Часто узкое место — не «медленный код», а конкуренция за ресурсы и неэффективная организация параллелизма. trace помогает это увидеть.

### 4. Анализ уровней в trace

1. STATS (pid 1) — общая статистика
- Goroutines — резкий рост и постепенный спад → массовый одновременный запуск.
- Heap — рост выделенной памяти; GC срабатывает при достижении порога.
- Threads — стабильно много потоков, что типично при большом числе горутин.

2. PROCS (pid 0) — нагрузка по логическим процессорам
- Каждый Proc N — логический поток исполнения, управляемый планировщиком Go.

**Что видно в деталях**

#### A. Массовый запуск горутин
Первые ~20 мс — множество тонких вертикальных штрихов на всех процессорах.
- → Горoutines стартуют почти одновременно, быстро блокируются (sleep) и конкурируют за CPU.

#### B. Работа GC
Около 15–20 мс — короткий розовый блок GC на всех процессорах.
- → Не источник проблем, но ранний запуск связан с массовыми аллокациями (стеки, структуры горутин).

#### C. Горутинa G1 — доминант
Длинные голубые блоки на разных процессорах.
- → Вероятно, main() или управляющая горутина (например, WaitGroup.Wait()), активно использующая ресурсы.

#### D. Частые контекстные переключения
Множество мелких прерывистых прямоугольников.
- → Короткие периоды исполнения, постоянная смена задач, высокие накладные расходы планировщика.

#### E. Низкая активность некоторых процессоров
Напр., Proc 6 и Proc 7 почти пусты.
- → Возможна неравномерная загрузка или недоиспользование CPU.

**Что делать дальше**
- Клик по блоку горутины (например, G1) → посмотреть stack trace, длительность, причину остановки.
- Вкладка Goroutine analysis → группы по функциям, состояние (Running/Runnable/Waiting).
- Профили:
  - Synchronization blocking — ожидание мьютексов и каналов.
  - Scheduler latency — полезно при большом числе горутин.

**Промежуточные выводы:**
- Много мелких блоков → частое переключение горутин.
- Нагрузка «вспыхивает» в первые 100–150 мс.
- Доминирующая горутина G1 управляет процессом.
- GC не мешает, но связан с массовыми аллокациями в начале.
- Несимметричная загрузка процессоров → возможная неэффективность планировщика.

**Что дальше**
- Откройте нужную горутину в trace, кликнув по её блоку, чтобы увидеть стек, длительность и причину остановки.
- Просмотрите профили внизу страницы — особенно Synchronization blocking, чтобы понять, где код ждёт мьютексы или каналы.

### 5. От симптома к решению

Проблемы производительности — это симптомы, а не баги: задержки, рост CPU, медленный отклик. Один и тот же симптом может иметь разные причины, поэтому первый шаг — наблюдать, а не «чинить».

**Системный подход:**
- Что именно происходит? (замедление, рост памяти, блокировки)
- Где проявляется? (какая функциональность)
- Какая нагрузка? (CPU-bound или IO-bound)
- Как выражается? (ожидание, конкуренция, узкие места)
- Что видно в профилях? (pprof, trace)

Правило: сначала понять, потом менять.

Оптимизация без данных = риск ухудшить ситуацию. Профили покажут, какие функции грузят CPU, где блокируются горутины и какие взаимодействия повторяются.

Решение — не всегда ускорение:
- ограничить параллелизм (пул воркеров);
- убрать лишнюю синхронизацию;
- перераспределить работу (кеш, предварительная обработка);
- изменить архитектуру.

**Пример:**
- Веб-сервис «залипает» при росте нагрузки, CPU не полностью загружен.
  - **Неэффективный путь** — наращивать горутины или буферы.
  - **Эффективный** — замерить, снять профили, понять, кто кого ждёт, и изменить архитектуру обработки.

**Отладка ≠ профилировка:**
- Отладка — «почему не работает».
- Профилировка — «почему работает так».

**Вывод**

Хороший инженер не просто пишет быстрый код, а понимает его поведение и управляет им.
