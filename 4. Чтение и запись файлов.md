# Чтение и запись файлов

Файл — это не просто значок на рабочем столе. Это структурированный набор данных, который хранится на диске и управляется операционной системой (ОС). 
 
**Файлы позволяют:**
- Сохранять результаты работы программы после её завершения
- Передавать данные между приложениями
- Хранить настройки, логи, медиафайлы и многое другое

В Go работа с файлами строится на двух китах:
- Пакет os — низкоуровневые операции (открытие, закрытие, чтение, запись).
- Пакет io — абстракции для ввода-вывода.

## Файл как абстракция: что такое os.File?
Когда вы открываете файл в Go, вы получаете объект типа *os.File. Это не просто «ссылка» на файл — это структура, которая содержит:

### Дескриптор файла
Дескриптор — это целое число, которое ОС присваивает открытому файлу. Это как номер столика в ресторане: официант (ОС) знает, что заказ (данные) относится к вашему столику (файлу).
- ОС использует дескрипторы для управления всеми открытыми файлами
- Каждая программа имеет ограничение на количество дескрипторов (зависит от ОС)
```go
file, _ := os.Open("data.txt")
fmt.Printf("Дескриптор: %d\n", file.Fd()) // Выведет число, например, 3 
```
### Позиция чтения/записи (offset)
Указатель, который определяет, с какого места в файле начнётся следующая операция чтения или записи.
- При открытии файла offset = 0 (начало файла)
- После чтения N байт offset сдвигается на N
- При записи offset перемещается в конец записанных данных
```go
file, _ := os.Open("data.txt")
buffer := make([]byte, 5)
file.Read(buffer) // Прочитает первые 5 байт, offset станет 5
file.Read(buffer) // Прочитает байты с 5 по 10 
```
### Метаданные
Объект os.File хранит информацию о файле:
- Путь к файлу
- Режим доступа (чтение, запись и т.д.)
- Права (например, 0644)

Права доступа — это набор разрешений, который определяет, какие действия могут выполнять разные категории пользователей с файлом. Обычно права задаются в восьмеричной системе (0XYZ), где:
- X — права владельца (User)
- Y — права группы (Group)
- Z — права остальных (Others)

Каждая цифра — сумма следующих флагов:
- 4 (r) — чтение: право открывать и просматривать содержимое файла
- 2 (w) — запись: право изменять или дописывать файл
- 1 (x) — выполнение: право запускать файл как программу

Например, 0644 означает, что владелец может читать и записывать файл (6 = 4 + 2), а группа и остальные имеют только право чтения (4).

## Жизненный цикл файла: от открытия до закрытия
### Открытие файла
Операция открытия файла — это диалог с ОС:
- Программа просит ОС: «Дай доступ к файлу "data.txt"».
- ОС проверяет права доступа и существование файла.
- Если всё в порядке — возвращает дескриптор.

Режимы открытия (флаги):
|Флаг|Описание|
|----|--------|
|os.O_RDONLY|Только чтение|
|os.O_WRONLY|Только запись|
|os.O_CREATE|Создать файл, если не существует|
|os.O_APPEND|Дописывать в конец файла|
|os.O_TRUNC|Очистить файл при открытии|
```go
// Открыть файл для записи, создать, если его нет, дописывать в конец.

file, err := os.OpenFile("logs.txt", os.O_WRONLY|os.O_CREATE|os.O_APPEND, 0644)
```
### Закрытие файла
Почему это важно?
- ОС ограничивает число открытых файлов для программы
- Незакрытые файлы могут привести к утечкам памяти
- Данные могут не записаться на диск (если используется буфер)
 
Используйте defer сразу после открытия:
```go
file, _ := os.Open("data.txt")

defer file.Close() // Закроет файл при выходе из функции 
```
**Что делает defer?**
- Откладывает выполнение file.Close() до завершения функции
- Гарантирует закрытие, даже если произошла ошибка
 
Когда вы пишете defer file.Close(), Go запоминает это действие и выполнит его автоматически в самом конце работы функции. Так вы сразу после открытия файла указываете, что его нужно закрыть, а сам код остаётся аккуратным: не нужно вставлять Close() в каждый блок обработки ошибок. В результате файл всегда закроется, даже если внутри функции произойдёт ошибка.

## Обработка ошибок: философия Go
### Паттерн(шаблон) обработки
В Go ошибки — не исключения, а часть возвращаемых значений. Это делает код предсказуемым.
 
Представьте, что вы готовите блюдо по рецепту. В языках с исключениями (например, Java или Python) это похоже на то, что если в рецепте что-то идёт не так — вы опрокидываете сковородку, раздаётся громкий сигнал, и вы сразу же перебрасываетесь к разделу «обработка аварийной ситуации» (бросаете «исключение»), а дальше код «ловит» его и решает, что делать.
 
В Go же кухня устроена иначе: каждый раз, когда вы кладёте в кастрюлю ингредиент, повар (функция) говорит вам не только «всё отлично, ингредиент добавлен» (возвращает основное значение), но и «дай знать, если что-то не так» (возвращает error). Вы сразу же проверяете этот сигнал:
```go
file, err := os.Open("data.txt")
    if err != nil {
// Обработка ошибки
    log.Fatal("Ошибка:", err)
} 
```
Это как если бы вы каждый раз после добавления ингредиента пробовали блюдо и говорили: «Всё в порядке — продолжаем», или «Ой, что-то горчит — остановимся и исправим», вместо того чтобы ждать, когда где-то в глубине рецепта треснет кастрюля и выключится плитка без объяснения причины. Такой подход делает процесс готовки (и ваш код) более прозрачным и понятным: вы видите каждую мелочь и решаете, как на неё реагировать, шаг за шагом.

### Типичные ошибки при работе с файлами
|Ошибка|Описание|
|------|--------|
|os.ErrNotExist|Файл не существует|
|os.ErrPermission|Нет прав доступа|
|io.EOF|Конец файла (при чтении)|

**Пример проверки:**
```go
if errors.Is(err, os.ErrNotExist) {
    fmt.Println("Файл не найден!")
}  
```
### Чтение файлов: от байтов до строк
**Что такое буфер?**
 
Буфер — это участок памяти, используемый для временного хранения данных при вводе-выводе. Когда мы читаем или пишем информацию на диск, данные сначала помещаются в буфер, а уже из него обрабатываются программой (или наоборот). Такой подход позволяет «сгладить» разницу в скорости между быстрыми операциями в памяти и относительно медленными операциями с диском, а также уменьшить число системных вызовов.
 
**Чтение всего файла (для маленьких файлов)**
```go
content, err := os.ReadFile("data.txt")
if err != nil {
    log.Fatal(err)
}
fmt.Println("Содержимое:", string(content)) 
```
Go под капотом обращается к операционной системе и запрашивает: «Прочитай весь файл целиком». ОС открывает файл, определяет его размер, создаёт буфер ровно нужного объёма и считывает в него все байты за одну операцию. Затем эти байты возвращаются вам в виде среза []byte. Преимущество такого подхода — минимум кода и никаких забот о циклах или ручном управлении буфером. Недостаток — когда файл действительно большой (десятки и сотни мегабайт), вы сразу же выделите под него столько памяти, сколько нужно, и рискуете «забить» оперативку. Поэтому os.ReadFile идеально подходит для небольших текстовых или конфигурационных файлов.
 
**Работа с путями к файлам **
 
Когда вы работаете с файлами в Go, важно правильно указывать путь: абсолютный (например, /home/user/data.txt) или относительный (data.txt, folder/file.txt). Чтобы создавать пути, которые одинаково работают на всех операционных системах, используйте пакет path/filepath. Он сам подставляет нужные разделители (/ или \) и упрощает обработку путей.
```go
filePath := filepath.Join("folder", "data.txt")  
```
Это особенно важно, если ваша программа должна работать и на Windows, и на Linux.
```go
filePath := filepath.Join("data", "data.txt")
content, err := os.ReadFile(filePath)
if err != nil {
    log.Fatal(err)
}
fmt.Println("Содержимое:", string(content))
```

### Построчное чтение (для больших файлов)
Используйте bufio.Scanner, чтобы не загружать весь файл в память:
```go
file, _ := os.Open("large_file.txt")
defer file.Close()
scanner := bufio.NewScanner(file)
for scanner.Scan() {
    line := scanner.Text() // Получить строку
    fmt.Println(line)
}

if err := scanner.Err(); err != nil {
    log.Fatal("Ошибка чтения:", err)
}    
```
Здесь bufio.NewScanner создаёт объект, который умеет «читать по строкам». Он сам заботится о разбиении потока байт на отдельные текстовые строки (по символу новой строки). Каждый вызов scanner.Scan() читает из файла до следующей строки, возвращая вам её текст через scanner.Text(). Вы держите в памяти только одну строку за раз, а не весь файл целиком. Это позволяет обрабатывать файлы размером в сотни мегабайт даже на машинах с ограниченным объёмом ОЗУ.
 
**Как работает буферизация?**
- При первом вызове scanner.Scan() bufio.Scanner запрашивает у ОС не по байту, а целый блок (обычно ≈4 КБ) и кладёт его в свой внутренний буфер
- Внутренний алгоритм разбивает этот блок по символам \n, отдаёт вам строки одну за другой и, когда буфер почти исчерпан, подгружает следующий блок
- Такой подход резко сокращает число дорогостоящих обращений к диску (системных вызовов) и делает чтение больших файлов намного быстрее

### Запись в файлы: от строк до JSON
**Запись всего содержимого**
```go
data := []byte("Hello, Go!")
err := os.WriteFile("output.txt", data, 0644)
if err != nil {
    log.Fatal(err)
} 
```
os.WriteFile сразу записывает все переданные байты в файл за одну операцию. Под капотом он создаёт (или перезаписывает) файл с указанными правами доступа (0644) и полностью сохраняет данные. Такой способ удобен для небольшой информации: например, сохранить текст, настройки или результат работы программы.
 
**Дозапись и буферизация**

Когда нужно многократно дописывать в файл (например, вести лог), лучше использовать буферизацию через bufio.NewWriter. Вместо каждой мелкой записи на диск данные сначала собираются в памяти. Только когда буфер заполняется или при явном Flush(), всё сразу записывается в файл. Это экономит ресурсы и ускоряет работу программы.
```go
file, _ := os.OpenFile("log.txt", os.O_WRONLY|os.O_CREATE|os.O_APPEND, 0644)
defer file.Close()
writer := bufio.NewWriter(file)
_ , err := writer.WriteString("Новая запись\n")
if err != nil {
    log.Fatal(err)
}
writer.Flush() // Сбросить буфер в файл! 
```
- os.OpenFile с флагами os.O_APPEND гарантирует, что каждая запись добавляется в конец файла, а не перезаписывает его
- bufio.NewWriter создаёт буфер (по умолчанию 4 КБ). Метод WriteString пишет данные в этот буфер, а не напрямую в файл
- Когда буфер заполняется, он автоматически сбрасывается (flush) в файл. Явный вызов Flush() необходим, чтобы гарантировать запись оставшихся данных перед закрытием файла или завершением программы
- Буферизация уменьшает число обращений к диску. Например, 1000 строк по 100 байт будут записаны 1–2 раза (по мере заполнения буфера), а не 1000 раз
- Важно: Если пропустить Flush(), часть данных может потеряться. Даже defer file.Close() не сбрасывает буфер — закрытие файла лишь освобождает дескриптор

### Практические советы
**Проверяйте существование файла**
```go
if _ , err := os.Stat("data.txt"); errors.Is(err, os.ErrNotExist) {
    fmt.Println("Файл не найден!")
}
```
Перед тем как работать с файлом, полезно проверить, существует ли он. os.Stat возвращает информацию о файле или ошибку, если файла нет. Это помогает заранее обработать ситуацию, избежать неожиданных сбоев при открытии или чтении.

**Работа с путями**
 
Используйте path/filepath для кросс-платформенных путей:
```go
//"folder/subfolder/file.txt" на Linux, "folder\subfolder\file.txt" на Windows

filePath := filepath.Join("folder", "subfolder", "file.txt") 
```
Функция filepath.Join правильно склеивает части пути, подставляя нужные разделители (/ или \) в зависимости от операционной системы. Это гарантирует, что ваш код будет работать одинаково на Windows, Linux и macOS.
 
**Права доступа:**
 
- 0644 — владелец может читать и писать, остальные — только читать
- 0755 — владелец имеет полный доступ, остальные — читать и выполнять

При создании файлов всегда стоит задуматься о правах: они определяют, кто сможет использовать ваш файл и каким образом. Неверные права могут привести к проблемам с безопасностью или ошибкам в доступе.
